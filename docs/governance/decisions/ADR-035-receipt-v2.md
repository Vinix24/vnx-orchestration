## 1. Context — the record, verified against the tree

VNX writes one NDJSON line per dispatch outcome to `.vnx-data/<project_id>/state/t0_receipts.ndjson` (ADR-005 canonical ledger). Two production paths write it (`docs/operations/RECEIPT_PIPELINE.md`):

- **Path 1 — governed dispatch envelope** (`governance_emit.emit_dispatch_receipt`, subprocess/multi-provider lanes).
- **Path 2 — report-on-disk → receipt processor** (`report_parser.py` → `append_receipt.py`, interactive/headless lanes).

Both funnel through `scripts/lib/append_receipt_internals/`. The documented schema (`docs/core/11_RECEIPT_FORMAT.md`, v1.0.0) is a flat bag: `dispatch_id`, `terminal_id`, `provider`, `model`, `status`, `completion_pct`, `risk`, `duration_seconds`, `token_usage`, `cost_usd`, `findings`, `pr_id`, `report_path`, `events_path`, `timestamp`, `recorded_at`, plus opt-in `prev_hash` (ADR-023). Enrichment (`enrichment.py`, `git_provenance.py`, `session_resolver.py`) adds `provenance{}`, `session{}`, `operator_id`/`project_id`/`orchestrator_id`/`agent_id`, `observability_tier`, `open_items_created`. Path 2's `report_parser.py::_build_enhanced_receipt` additionally regex-extracts `tags`, `root_cause`, `dependencies`, `recommendations`, `metrics{performance,quality,business}`, `validation{tests_passed,tests_failed,quality_gates}`, `pattern_count`, `prevention_rules`, `quality_context`, `used_pattern_hashes`, `legacy_format`/`missing_fields`, and a `confidence` score.

Three measured pain points ground this redesign:

1. **The record is too thin to decide on, and also too wide to read.** T0/operator must decide accept/investigate/reject per dispatch. Today that requires opening the report, because no field states a decision — `status` is a raw outcome string, not a verdict, and the fields that *would* support a verdict (test results) are Path-2-only, regex-extracted, and undocumented (`validation{}` never made it into `11_RECEIPT_FORMAT.md`). A genuine subset of the record is pure write-side noise — re-verified field-by-field against current `main`, not assumed: grepping for `.get('root_cause')`, `.get('dependencies')`, `.get('metrics')`, `.get('prevention_rules')`, `.get('used_pattern_hashes')`, `.get('legacy_format')` as *receipt-field reads* turns up **zero** consumers anywhere outside `report_parser.py`'s own write side. `tags` belongs in this dead group too, despite an earlier draft of this ADR claiming otherwise: `t0_intelligence_aggregator.py:466` does assign a `tags` key onto its own enriched-receipt copy, but the value comes from `reports_dir/index.ndjson`'s tag cache (`_load_report_tags`), never from `report_parser.py:700`'s regex-extracted receipt field — so that field, too, has no real reader; the earlier claim conflated "writes a same-named key" with "reads this field." Four other fields this same earlier draft *also* assumed dead turned out not to be, once each was individually re-grepped rather than bucketed with the rest: `recommendations` (`scripts/generate_t0_recommendations.py:473`), `quality_context` and `pattern_count` (`scripts/check_intelligence_health.py:151`, `:156`), and `missing_fields` (`scripts/backfill_headless_receipts.py:238`, `:259`, a receipt-backfill utility that parses `t0_receipts.ndjson` lines directly) all have live, executable readers. §3.2/§3.3 corrects the record: none of those four are removed in v2. The clearest still-dead case is `confidence`: `_build_enhanced_receipt` computes it as `min(0.98, 0.50 + field_count * 0.08)` — a fixed arithmetic function of how many *other* optional fields happened to extract, not a measurement of anything about the dispatch. It reads as a confidence score. It is not one.
2. **A failure flag with no consequence is noise, not signal.** `report_contract_invalid` was flagged as VNX's #1 governed failure at 2274 occurrences. Root-cause analysis (memory `report-contract-invalid-is-dead-benchmark-noise`, 2026-07-09) found 100% are from June 2026 benchmark/smoke dispatches, zero since 2026-06-27, and the validator only ever logs a failure — there is no `report_contract_valid` counter-event and no time window, so an unwindowed dashboard shows the same 2274 as "live" forever. This is the general failure mode pillar 4 closes: a warning with no destination doesn't get fixed or explicitly retired, it just accumulates as ambient noise. (Per that same memory, this ADR does **not** rebuild the markdown→JSON report pipeline to chase that number — see §8 non-goals. The number is this ADR's motivating *example* of the destination problem, not its target.)
3. **Terminal/session state is stamped onto an immutable ledger line, and delivery is a push.** `session_resolver.py` resolves `session_id`/`terminal`/`model`/`provider` via a five-step priority chain (report-provided → per-terminal file → env var → provider file → `"unknown"`) and `enrichment.py` writes the result as a `session{}` object on every receipt — that is live coordination state (the kind ADR-005 assigns to `runtime_coordination.db`), duplicated onto an append-only audit line at write time. Separately, `rp_delivery.sh` pushes each receipt into the T0 tmux pane via `load-buffer`/`paste-buffer`/`Enter` (an outbox-and-retry pattern hardened as recently as #1178, 2026-07-16, after a stray-paste incident). `VNX_RECEIPT_T0_PUSH=0` already exists as a kill switch and is the operator's current default post-incident — this ADR designs what replaces the push it is silencing, not merely how to keep silencing it.

## 2. Decision

Receipt v2 is an **additive schema revision** on the same ADR-005 ledger, gated by a new `schema_version` field, delivering four structural changes:

1. A dense, deterministically-computed **`verdict{}`** object so accept/investigate/reject is readable off one field, never inferred from opening the report (§3).
2. **Terminal/session state removed** from the receipt body down to a stable pointer; the state itself lives in `runtime_coordination.db`, where ADR-005 already assigns it (§4).
3. A **pull query interface** (`receipt_query.py`) replacing the tmux-pane push; T0 and tooling read the ledger on their own cadence (§5).
4. Every warning gets an **enforced destination** — OI-promotion (confirmed or, when the OI store itself fails, durably pending), aggregated count, or an explicit reason for dropping — with nothing left to silently accumulate (§6).

Append-only NDJSON stays canonical (ADR-005 unchanged). Receipts stay local per project (ADR-026 unchanged, restated as a hard constraint in §8). Hash-chain/epoch machinery (ADR-029/034) is untouched — v2 is a payload-shape change, and `verify_chain` hashes serialized line content, not a fixed field set (§7).

### 3.1 New fields

| Field | Type | Why |
|---|---|---|
| `schema_version` | int | Reader-dispatch key. `2` from cutover forward; absent (or `1`) means the legacy shape below. Same convention ADR-023/029 already use for `prev_hash`/epoch markers: field presence, not a file-level version marker, because the ledger is line-oriented and mixed-version by construction (§7). |
| `verdict.decision` | enum: `accept` \| `investigate` \| `reject` | The field this whole ADR is for — the one-line answer to "do I need to open the report." Computed by a pure function (`receipt_verdict.py`, §9 PR-1) from `status` + `verification` + `warnings[]`, never free-text from the worker and never an LLM judgment call — a fixed rule table is auditable and unit-testable, a paragraph of prose isn't. Rule: `reject` on a hard-failure `status` (`failed`/`error`/`blocked`/`timeout`/`contract_invalid`) or any `warnings[]` entry with `severity: "blocker"` — a condition `compute_verdict` can key on `severity` alone, without separately re-checking whether the entry was "really" tracked, because the append validator now guarantees (§6.1's stateless `severity: "blocker" ⇒ requires_tracking: true ⇒ destination ∈ {oi, oi_pending}` invariant, r3 HIGH-3) that a blocker can never reach a durable receipt silently downgraded to `counted`/`dropped`; `investigate` when `status` claims success but `verification` is absent/incomplete (no test evidence for a claimed-done dispatch), `verification.method == "pending-report"` (§3.1.1), or a warning has `destination: "oi_pending"` (§6.1, §6.4 — promotion requested but not yet confirmed is not a clean accept); `accept` when `status` claims success, `verification` shows `tests_failed: 0` with `tests_run > 0`, and no blocker warnings and no `oi_pending` warnings. `method: "n/a"` (doc-only dispatch) can also reach `accept`, but **only** under the doc-only invariant below — it is not a free pass. |
| `verdict.reason` | string | One line, human-composed from the same inputs the rule table used — the "why," not a restatement of the decision. |
| `verdict.evidence_complete` | bool | `false` when `verification.method` is `"unknown"`, `"none_claimed"`, or `"pending-report"` — lets a digest distinguish "we checked and it's clean" from "we don't actually know," which today's silent absence conflates. |
| `verification{}` | object | Promotes Path 2's existing (but undocumented, ad-hoc) `report_parser.py::extract_validation` output to a canonical, documented field, and extends it to Path 1 (§3.1.1 — the two Path-1 sub-paths need different treatment because they write report and receipt in opposite order). `method` is one of `pytest`\|`manual`\|`none_claimed`\|`n/a`\|`pending-report`\|`unknown`; `unknown` is the honest failure mode when extraction can't find a verification section at all — an explicit "we don't know" beats the current silent absence. This ADR does **not** replace markdown reports with a JSON contract (§8) — it reuses the existing regex extractor and gives its output a stable name and a schema slot. |
| `verification.pr_ref` | string \| null | The PR/commit this verification event is anchored to, first-class rather than something a reader infers by cross-referencing `pr_id`/`provenance.git_ref` — those can legitimately differ from what was actually *verified* (e.g. `pr_id` updated post-hoc, `provenance.git_ref` captured from a dirty working tree). Explicit evidence pointer, per BLOCKING-2. |
| `verification.push_verified` | bool | Whether the writer actually confirmed (e.g. `git ls-remote` against the target branch, per this dispatch's own report contract) that the referenced commit reached the remote — distinct from merely claiming it did. `false`/absent is not an error; it just means `verdict.evidence_complete` can't credit push-confirmation, only local test evidence. |
| `verification.spec_deviation` | string \| null | `null` asserts "this dispatch delivered exactly what the spec asked for, nothing more, nothing withheld." Non-null documents *how* it deviated (scope cut, extra fix bundled in, spec ambiguity resolved a specific way). **Must be explicitly set, never defaulted**, whenever `method` is `n/a` or `none_claimed` — an accept with no test evidence and no deviation attestation is exactly the accept-without-evidence gap BLOCKING-2 closes. |
| `warnings[]` | array | Replaces the `quality_advisory.t0_recommendation.open_items[]` shape one-for-one — see §6 for the full mechanism. Each entry: `{code, severity, message, destination, oi_id, reason, requires_tracking}`. |
| `session_id` | string | The **only** session-shaped field left on the receipt — a pointer, not a copy. See §4. |
| `event_type` | string, required + non-empty | The completion-detection field the existing Path-2 validator already requires and live consumers already filter on — promoted to an explicit, required-and-non-empty, documented field on **both** write paths for the first time in v2. **Not** a closed enum: revision 2 of this ADR specified one, and r3 BLOCKING-1 rejected that shape as fragile against an evolving fabric (§3.2.1, §11). For `schema_version >= 2`, the legacy `event` alias is no longer accepted as a substitute for a missing `event_type` — v1 lines keep the alias, v2 does not (r3 HIGH-2, §3.2.1). See §3.2.1 for the field-value survey (kept as evidence for the enum's rejection, not as an allow-list) and the Path-1 stamping gap this closes (r2 BLOCKING-1 — distinct from r1's BLOCKING-1, the hash-chain writer-contract gap, §7.1). |

#### 3.1.1 Path 1 verification — the two sub-paths write in opposite order

`RECEIPT_PIPELINE.md` documents Path 1 as two sub-paths with **opposite** report/receipt write ordering, and `emit_dispatch_receipt`'s signature (`scripts/lib/governance_emit.py:53-74`) carries no verification-related parameter at all today — confirming BLOCKING-2's premise that Path 1 has no test-evidence ingang, on either sub-path, as things stand:

- **Envelope sub-path** (`dispatch_envelope.py`, subprocess/tmux lane): `emit_unified_report` runs, **then** `emit_dispatch_receipt`. The markdown report already exists on disk by the time the receipt is written. v2 threads the *same* `report_parser.py::extract_validation` regex extractor Path 2 already uses into this call, reading `report_path` and populating `verification{}` from it — no new extraction mechanism, no new report format (§8 non-goal preserved).
- **Multi-provider sub-path** (`provider_dispatch.py`): `emit_dispatch_receipt` runs **first**; `report_path` is a precomputed deterministic string, the file doesn't exist yet (`provider_dispatch.py:480-482`). There is nothing to extract. v2 stamps `verification.method: "pending-report"` explicitly — never a silent blank — which `compute_verdict` reads as `investigate`, `evidence_complete: false`. This is **not** later backfilled: ADR-005 append-only forbids rewriting the line, and `report_parser.py`/`append_receipt.py` (Path 2's machinery) is never invoked for a Path-1 dispatch, so no second write ever arrives to fill the gap. This is a permanent, honest property of this sub-path's write ordering — a reader who wants corroborating evidence follows `report_path` themselves. Chosen over inventing a receipt-amendment/second-write mechanism, which would be new ledger-append machinery beyond what this ADR's non-goals (§8) already rule out, and over silently downgrading write-ordering to match Path 2 (out of scope — `provider_dispatch.py`'s receipt-first ordering exists for its own idempotency reasons, untouched here).

**Doc-only invariant.** `method: "n/a"` may resolve to `verdict.decision: "accept"` only when `provenance.diff_summary.paths` (§3.2 — a new list this ADR adds to `diff_summary`, populated from a `git diff --name-only` call alongside the existing `--shortstat` call, same gate, no new git invocation class) shows **every** changed path under `docs/**` or matching `*.md`. If any changed path falls outside that set, `compute_verdict` forces `investigate` regardless of `status` — a code change claiming "n/a" test evidence is exactly the accept-without-report failure mode BLOCKING-2 is about, and this makes the invariant mechanically checked at verdict-compute time, not asserted by convention.

**Missing-`paths` is fail-safe, never fail-open (r2 HIGH-4 — distinct from r1's HIGH-4, the `by-track` query field gap, §5.2).** `paths` is populated by §9 PR-4 (moved up from PR-5 — see the reordering note in §9), the same PR that first wires `compute_verdict` live. That closes the *scheduling* gap, but a live receipt can still legitimately lack `paths` after PR-4 too — a receipt written by a lane/version that predates PR-4 in a mixed-rollout window, or a `git_provenance.py` capture that itself failed (git command errored, dirty-tree edge case) and left `diff_summary` without the new subkey. `compute_verdict` treats `paths` being absent, `null`, or an empty list, under `method: "n/a"`, as **unproven** doc-only-ness, not as a vacuously-true "every path" check — the rule is `investigate`, never `accept`, whenever `paths` cannot be inspected. Absence of evidence is not evidence of doc-only. This is the same fail-closed posture §6.1 already applies to `warnings[]` shape violations, applied here to a missing-evidence case instead of a malformed one. T35 (§10) is the regression guard, distinct from T23 (which covers `paths` present but containing a non-docs path).

### 3.2 Fields kept, unchanged

`dispatch_id`, `terminal_id`, `provider`, `model`, `status`, `duration_seconds`, `token_usage`, `cost_usd`, `pr_id`, `report_path`, `events_path`, `timestamp`, `operator_id`, `project_id`, `open_items_created` (now derived — see §6), `prev_hash` (opt-in, ADR-023, untouched). `provenance{git_ref, branch, is_dirty, diff_summary, in_worktree, worktree_path}` is kept **by name** — not renamed to `diff` as an earlier draft of this design considered — because `scripts/lib/receipt_provenance.py::find_receipt_by_commit` (line ~773) reads `entry.get("provenance", {}).get("git_ref")` today; renaming the key would silently break a live consumer. `provenance.diff_summary` gains one new subkey, `paths: string[]` — the changed-file list, from a `git diff --name-only` call alongside the existing `--shortstat` call in `git_provenance.py::_build_git_provenance` (same `is_dirty` gate, no new git invocation class) — needed by the doc-only invariant (§3.1.1). Only the object's dead subkeys are trimmed (§3.3).

**Kept, corrected from an earlier draft** (re-verified by grep against current `main`, not assumed dead — HIGH-5): `recommendations` (read by `scripts/generate_t0_recommendations.py:473`, feeding T0 recommendation intelligence), `quality_context` and `pattern_count` (read by `scripts/check_intelligence_health.py:151` and `:156`, feeding the intelligence-health coverage counter), `missing_fields` (read by `scripts/backfill_headless_receipts.py:238`, `:259`, a receipt-backfill utility that parses `t0_receipts.ndjson` lines directly and filters this field). None of the four is dropped in v2; §3.3 no longer lists them.

#### 3.2.1 `event_type` — made explicit, required-and-non-empty (not closed-set), schema-version-gated against the `event` alias, and bound to both write paths (r2 BLOCKING-1, r3 BLOCKING-1, r3 HIGH-2)

**Verified gap.** `_validate_receipt` (`append_receipt_internals/validation.py:53-68`) already requires `event_type` (or its `event` alias) as a hard precondition — `not event_name` raises `missing_required_key` before the append proceeds — and does so *today*, on Path 2. Multiple live consumers key directly off a specific value: `outcome_signals.py:113` and `:118` branch on `event_type == "task_complete"` paired with `status == "failed"`/`"success"` respectively (both outcomes share the *same* `event_type`, with `status` carrying the verdict — the convention v2's Path-1 fix below follows); `pr_discovery.py:89` and `unified_state_manager.py:339` filter the same way; `retroactive_backfill.py:117` and `t0_intelligence_aggregator.py:343` count `task_complete` receipts for throughput metrics. None of this is new to v2 — it is the *undocumented* status quo `11_RECEIPT_FORMAT.md`'s flat-bag field list (§1) omits entirely.

What v2 changes: `emit_dispatch_receipt` (Path 1, `governance_emit.py:106-158`) builds its receipt dict from `dispatch_id` through `injection_reconstructs` today and **stamps no `event_type`/`event` key at all** — confirmed by reading the full dict-construction block. Path 1 has never gone through `_validate_receipt` (§6.1's original bypass finding, r1), so this has been silently harmless so far. §7.1 fixes that bypass by routing Path 1 through the same shared, validating append primitive Path 2 uses — and the moment that lands, every Path-1 write hits the existing `not event_name` check and is **rejected outright**, because nothing upstream of §7.1 ever taught `emit_dispatch_receipt` to set the field. This is the exact failure scenario named: fixing the validator bypass without also fixing the writer turns governed receipts into ones that never land.

**The fix, in three parts.**

1. **`emit_dispatch_receipt` stamps `event_type: "task_complete"` unconditionally**, added as a plain key in the receipt-dict construction (`governance_emit.py`, alongside `dispatch_id`/`status`/etc.) — not conditioned on `status`. Both call sites (`dispatch_envelope.py:509`, `provider_dispatch.py:640`) invoke `emit_dispatch_receipt` **exactly once per dispatch, at the terminal outcome** (verified — neither call site loops or calls it for intermediate states like `task_started`), passing `status` as `"success"`/`"failed"`/`"timeout"`/etc. as a separate field. This mirrors the existing `outcome_signals.py` convention exactly: `task_complete` marks "the dispatch reached a terminal outcome," `status` carries which outcome — so `task_complete` is correct for a failed dispatch too, not only a successful one. `task_failed`/`task_timeout` remain live, distinct values elsewhere in the ledger (e.g. `gate_stack_resolver.py::build_rejection_receipt` stamps `task_failed` for a pre-dispatch tier-admission rejection), but those are a different call path — a dispatch that never reaches `emit_dispatch_receipt` at all — not a case Path 1's fix needs to distinguish.
2. **The validator's `event_type` check is required-and-non-empty — deliberately not a closed enum (r3 BLOCKING-1, supersedes revision 2's design).** Revision 2 of this ADR specified an explicit allow-list, `LEGAL_EVENT_TYPES`, rejecting any value outside a compiled inventory. Re-reviewed by codex and rejected as structurally wrong against an evolving fabric: a fresh grep for this revision (not revision 2's) turns up 50+ distinct `event_type` string literals written across the tree — governance events via `emit_governance_receipt` (`scripts/lib/governance_receipts.py:16-28`, e.g. `event_type="review_gate_result"` from `gate_result_parser.py:93,206`), `report_contract_invalid` (`report_to_receipt_converter.py:266`, `report_parser.py:672`), `pr_merged` (`backfill_pr_merged_receipts.py:211`), `context_rotation_continuation` (`context_rotation.py:751`), `phantom_guard_error` (`phantom_guard.py:378`), and more — with no central registry constraining what a future PR adds. A closed enum against that reality doesn't fail once at design time; it fails the next time a PR introduces a legitimately new event type, silently rejecting that receipt at the exact moment PR-2's tightened validator goes live — the opposite of pillar 1's goal, which is that nothing should silently fail to land. v2 therefore keeps the check at what BLOCKING-1's original ask was in spirit — "this event has *a* value, not none" — plus one addition beyond today's v1 behavior: a lightweight format guard, `^[a-z][a-z0-9_]*$` (lowercase, snake_case), applied to whichever value the check resolves. This catches transcription/typo garbage (`"TaskComplete"`, `"task complete"`, a value that's only whitespace) without asserting membership in any fixed set — every value in the survey below already conforms to this shape, so the guard costs nothing against real traffic. **The inventory revision 2 compiled as `LEGAL_EVENT_TYPES`'s membership set is kept below, but repurposed** — not as an allow-list the validator checks membership against, but as the evidentiary survey grounding BLOCKING-1's rejection of that shape, proof that 40+ live values already exist for the *current* fabric before a single future value is even considered: `task_started`, `task_complete`, `task_failed`, `task_timeout`, `task_blocked`, `dispatch_sent`, `dispatch_ack`, `ack` (today's `DISPATCH_REQUIRED_EVENTS`, `validation.py:14-23`); `state_mutation` (today's `STATE_MUTATION_EVENTS`, `validation.py:25`, written by `state_mutation.py::emit_state_mutation`); `subprocess_completion` (the tmux-interactive-lane completion receipt — `tmux_interactive_dispatch.py:917,1718`, `pr_enforcement.py:122`, `phantom_guard.py:335` — including this dispatch's own completion-protocol receipt); the legacy completion aliases `task_completed`, `completion`, `complete` (`validation.py::_is_completion_event`, `payload.py:449`'s `_maybe_trigger_state_rebuild`); `dispatch_promoted`, `dispatch_started` (`payload.py:449`'s same trigger set); `review_gate_request` (`validation.py:42`) — plus the governance/report-contract survey named above, none individually re-grepped to a line number a second time, cited to make the survey's point (this list is not, and cannot be, exhaustive) rather than to extend the now-dropped allow-list. One value cited as illustrative context in codex's r3 BLOCKING-1 finding, `codex_final_gate`, was checked against the tree for this revision and found **not** to be an `event_type` literal anywhere — only a gate/module name (`gate: str = "codex_final_gate"`, `codex_final_gate.py:453`) — and is deliberately excluded from this survey rather than carried over unverified. §9 T29 is rewritten accordingly (§10): it no longer asserts a compiled set is *exactly* what's accepted; it asserts the opposite — a never-before-seen value is accepted so long as it's present, non-empty, and snake_case.
3. **For `schema_version >= 2`, the `event` alias is rejected — `event_type` itself is the only legal key (r3 HIGH-2).** Today's validator (`validation.py:62`) accepts `receipt.get("event_type") or receipt.get("event")` unconditionally, and that fallback exists for a real reason: v1 lines written before this field had a canonical name legitimately used `event`, and a v1-shaped reader/validator must keep tolerating that shape forever (ADR-005 append-only — no rewriting the past). But nothing in that check keys off `schema_version`, so a v2 write — `schema_version: 2`, `event: "task_complete"`, no `event_type` key at all — passes today's check, and would keep passing under point 2's required-and-non-empty fix too, landing durably and invisible to every `event_type == "task_complete"` consumer (`outcome_signals.py:113,118`, `pr_discovery.py:89`, `unified_state_manager.py:339`) — the same class of failure r2 BLOCKING-1 closed for Path 1's *missing* key, reappearing here for either path's *aliased* key. The fix is a schema-version-gated branch in the same check: when `schema_version` is present and `>= 2`, only `receipt.get("event_type")` is consulted — a v2 record with `event` set but `event_type` absent or empty is **rejected**, `event` is not read as a fallback at all. When `schema_version` is absent or `1`, both keys are still consulted exactly as today (`event_type` first, `event` as fallback) — v1's alias tolerance is untouched. This is a one-line branch on a field the validator already has in hand, not new validator machinery.

**Binding to §7.1.** §9 PR-3 (the shared-append-primitive extraction) now explicitly includes point (1) above as part of unifying Path 1 onto Path 2's writer — the fix is not deferred to a later PR, because PR-3 is precisely the PR that first exposes Path 1 to the validator. See §7.1 and §9.

## 4. Terminal-state out — where it goes instead

`session{}` (session_id, terminal, model, provider, `captured_at`, status/error) is `session_resolver.py`'s live resolution of *which terminal/session is currently running*, re-derived and re-stamped on every single receipt. That is exactly the class of information ADR-005 already assigns to `runtime_coordination.db` ("leases, heartbeats, incident log" — SQLite projection, not the ledger). Stamping it onto an append-only line means the ledger carries a snapshot of mutable state that was already stale by the time the next receipt was written, and duplicates ~40 bytes of near-constant data across thousands of lines for no query benefit today's tooling actually exercises.

v2 keeps exactly one field, `session_id` — a stable string, resolved once via the existing `session_resolver.py` priority chain, written as-is. Anything that needs the *rest* of the session's state (model, provider, terminal, liveness) looks it up in `runtime_coordination.db` by `session_id` at read time. This is not a new lookup path to build from scratch: `runtime_coordination.db` already tracks per-session/heartbeat rows (`scripts/lib/runtime_reconciler.py`, `scripts/lib/workflow_supervisor.py` both resolve against it today); this ADR's implementation PR (§9, PR-5) wires the receipt-side removal, not a new DB schema.

`events_path` is left exactly as-is and cited as the pattern this generalizes: it already replaced a filename-convention link with an explicit pointer (PR #843, ADR-005 amendment 2026-06-13) rather than carrying the event stream's content inline. `session_id` does the same thing for session state.

The same principle governs §5.2's `by-track` decision: track membership is `dispatches.track`-owned relational state, keyed by `dispatch_id` — a receipt does not grow a `track_id` field to carry a copy of it, for the same reason it does not grow the rest of `session{}` back.

## 5. Pull model — interface, and how it relates to WS7

### 5.1 What exists today, verified

Two independent things currently do part of what a pull interface needs, and neither is wired into current `main`:

- **`receipt_provenance.py`** (main, live) already implements dispatch-indexed and commit-indexed queries — `find_receipts_by_dispatch`, `find_receipt_by_commit`, `find_commits_by_dispatch`, `provenance_summary_for_dispatch`, `batch_provenance_summary`. These are linear NDJSON scans, no index. There is **no** PR-indexed or track-indexed query, and no since-timestamp query — genuine gaps.
- **`receipt_pull.py`** (parked, branch `feat/receipt-mailbox-delivery`, commit `54089155`, 2026-06-21; follow-up wiring commit `24f71d22` same day) implements the cursor half: byte-offset read-then-advance, advances only past complete (newline-terminated) lines so a concurrent append's partial trailing line is never consumed early, resets to 0 on a truncated/rotated ledger, `--seed-now` sets the cursor to EOF to skip the backlog without deleting it, `--peek` reads without advancing. This was built against a 4-model panel (GLM-5.2, DeepSeek-v4-pro, Kimi, Opus) that validated the pull *direction*. The branch is **1,509 commits behind `origin/main`** as of this writing and predates the #1178 push-hardening work by nearly a month — it is not mergeable as-is, and this ADR does not attempt to.

Per the operator's explicit instruction not to duplicate WS7: this ADR **absorbs** `receipt_pull.py`'s cursor algorithm (proven correct by its own logic and the panel review) and **extends** `receipt_provenance.py`'s query functions, rather than re-deriving either from scratch or resurrecting the stale branch wholesale.

### 5.2 `receipt_query.py` — the pull interface

```
scripts/receipt_query.py pull       --state-dir <dir> [--cursor-file <path>] [--seed-now] [--peek] [--json]
scripts/receipt_query.py by-dispatch <dispatch_id> --state-dir <dir> [--json]
scripts/receipt_query.py by-pr      <pr_id>        --state-dir <dir> [--json]
scripts/receipt_query.py by-track   <track_id>      --state-dir <dir> [--json]
scripts/receipt_query.py since      <ISO8601>        --state-dir <dir> [--json]
scripts/receipt_query.py digest     --state-dir <dir> [--window 24h] [--json]
```

- **`pull`** — the tick primitive. Signature and semantics are `receipt_pull.py`'s, reimplemented fresh against v2's schema and current `main` (not a branch resurrection): byte cursor in `receipt_pull_cursor.json`, read-then-advance, partial-trailing-line-safe, truncation-reset, `--seed-now`/`--peek` unchanged. This is what a T0 cycle calls at the start of its turn instead of waiting for a pane paste.
- **`by-dispatch`** — thin wrapper over the existing `receipt_provenance.find_receipts_by_dispatch`. No reimplementation.
- **`by-pr`, `since`** — new, but same linear-scan-with-predicate approach `find_receipts_by_dispatch` already uses (no new index or SQLite projection in this phase — see §8 non-goals; at the fleet's current scale, ~8,000–13,000 lines per project ledger, an O(n) scan with early field-match is not a measured bottleneck. If it becomes one, ADR-005 already permits a SQLite projection downstream of the ledger; that is a future ADR, not this one). Both predicates (`pr_id`, `timestamp`) are already fields on every receipt line, so a linear scan can actually filter on them.
- **`by-track`** — **not** a linear NDJSON scan (HIGH-4: the receipt record carries no `track_id` field to filter on, so a linear-scan-with-predicate approach cannot work here at all — there's nothing on the line to match). This ADR does not add `track_id` as a receipt field either (§4's cross-reference, §8): track membership is `dispatches.track`-owned relational state in `runtime_coordination.db`, the same class of thing session state was for §4, and duplicating it onto the immutable ledger reintroduces the exact anti-pattern pillar 2 just removed. Instead, `by-track` is a two-step join, reusing existing code rather than adding a new one: (1) `SELECT dispatch_id FROM dispatches WHERE track = ? AND project_id = ?` — the identical query `tracks.get_recent_receipts` (`scripts/lib/tracks.py:814-829`) already runs today; (2) for each `dispatch_id`, `receipt_provenance.find_receipts_by_dispatch` — the existing dispatch-indexed lookup `by-dispatch` already wraps. No new index, no new SQLite table, no receipt-shape change; `by-track` composes two functions that already exist on main.
- **`digest`** — the pull-model's answer to `rp_delivery.sh`'s old push-side digest (`_rpd_deliver_digest`, #1178): verdict counts (`accept`/`investigate`/`reject`) over the window, plus the top `warnings[]` codes sitting at `destination: "counted"` with their running totals — this is where "N warnings without an OI-link" (§6) becomes visible without a per-instance ping.

Every subcommand must handle a mixed v1/v2 ledger — `schema_version` absent lines report `verdict: "unknown"` in `digest`'s counts rather than crashing or being silently miscounted as any real verdict (T17, §10).

### 5.3 Retiring the push

`rp_delivery.sh`'s pane-paste path (`_deliver_receipt_to_t0_pane`, `_rpd_deliver_digest`, the pending/processed outbox) is **superseded, not deleted in this PR**. The write side it depends on — `send_receipt_to_t0`'s write-first-then-attempt-delivery pattern — already cleanly separates "the ledger line is durable" from "T0 was notified"; that separation is exactly what makes retiring the *notification* half safe without touching the *durability* half. `VNX_RECEIPT_T0_PUSH` (already effectively OFF per the operator's 2026-07-16 post-incident default) stays as the transition escape hatch through the rollout in §9, and is removed — not merely defaulted off — only once `receipt_query.py pull` is confirmed wired into the T0 cycle fleet-wide (mirroring the intent of the parked `24f71d22` skill-rule wiring commit, redone against current `main`). Removing the kill switch entirely is a follow-up PR, not part of this ADR's decomposition.

## 6. Warning-destination rule — the mechanism, not the intention

Every `warnings[]` entry carries a **mandatory** `destination` field, computed by the writer (never left to the emitting caller), before the receipt is durably appended. It also carries a `requires_tracking` boolean, computed by the same engine in the same pass — see §6.1's matrix for why this field exists.

```json
{
  "code": "report_contract_invalid",
  "severity": "warn",
  "message": "Summary section missing (## Summary heading not found)",
  "destination": "counted",
  "oi_id": null,
  "reason": null,
  "requires_tracking": false
}
```

A `severity: "blocker"` entry whose OI-promotion could not be confirmed before the receipt had to be appended:

```json
{
  "code": "worker_permission_violation",
  "severity": "blocker",
  "message": "worker wrote outside its declared file-write scope",
  "destination": "oi_pending",
  "oi_id": null,
  "reason": "open_items_manager.add_item_programmatic raised: [Errno 11] store lock held by pid 4821",
  "requires_tracking": true
}
```

### 6.1 The legal destinations

1. **`"oi"`** — promoted to a tracked open item, `oi_id` set. Fires when `severity: "blocker"`, or when `severity: "warn"` **and** the same `code` has recurred at or above a rolling-window threshold (default 3, configurable) for the same dispatch-adjacent scope (skill/terminal). This generalizes today's `_register_quality_open_items` (`scripts/lib/append_receipt_internals/quality.py`), which already does exactly this for the single case of `quality_advisory.t0_recommendation.open_items[]` — dedup key `qa:{check_id}:{file}:{symbol}`, `open_items_manager.add_item_programmatic`. v2 makes this the **general** path for any warning class, not a special case for one advisory type.
2. **`"oi_pending"`** — OI-promotion was **attempted** (the entry qualified for `"oi"` under rule 1) but could not be **confirmed** before the receipt had to be durably appended — `open_items_manager.add_item_programmatic` raised, or the store was unreachable/locked. Legal **only** with a non-null `reason` describing the failure, and **only** with `oi_id: null` — the open item does not exist yet, so there is nothing to point at. See §6.4 for the full lifecycle and the follow-up obligation this destination carries; this is r2 BLOCKING-2's fix (distinct from r1's BLOCKING-2, the Path-1 verification-evidence gap, §3.1.1).
3. **`"counted"`** — aggregated into a rolling per-`code` counter, surfaced via `receipt_query.py digest`'s "N warnings without an OI-link" tally, never individually re-surfaced. This is the destination `report_contract_invalid`-class noise gets from day one of v2: visible, windowed, boundable — not a silent 2274-and-climbing dashboard artifact. Legal only when `requires_tracking: false`.
4. **`"dropped"`** — legal **only** with a non-null `reason` **drawn from a defined allow-list of drop reasons** (HIGH-3 tightens this from "any non-null reason" to "a reason from a closed set"), and only when `requires_tracking: false`:
   - `retired_check` — the check/rule that produced this warning code has been permanently removed; no further occurrences are expected.
   - `duplicate_of_blocker` — a warn-severity restatement of a blocker already tracked under a different `code` on the same dispatch; dropping avoids double-counting the same root cause.
   - `superseded_by_code` — replaced by a newer check `code` covering the same condition, during that code's sunset window.
   - `operator_acknowledged_noise` — an operator, not the writer, has explicitly marked this specific code as accepted noise; `reason` must carry an attributable, dated decision reference, not a free-form excuse invented by the writer at append-time.

   A `reason` string outside this set is rejected exactly like a null `reason` was in r1 — the append fails closed either way, it does not silently accept an undocumented or unlisted drop. This is pillar 4's literal enforcement point: a warning with nowhere legitimate to go cannot exist in a v2 receipt.

**`requires_tracking` — the severity × destination matrix, enforced structurally (r2 HIGH-3, no r1 collision).** r1's reject list blocked missing/unknown `destination`, but not a `severity: "blocker"` entry silently downgraded to `"counted"` — the exact engine-bug scenario HIGH-3 named, extending the same reject list r1's BLOCKING-3 first bound to both write paths: a bug stamps a new blocker as `counted`, the entry has no `oi_id`, and the blocker gets no tracked open item, with nothing rejecting it. The validator cannot close this by independently recomputing whether an entry *should* have required tracking — the `severity: "warn"`-recurred-at-threshold half of the rule needs a rolling-window read against dispatch-adjacent history, which is exactly the kind of stateful lookup this ADR keeps out of the append-time validator (§7.1 already keeps the shared primitive's lock scope to hashing/writing, not arbitrary I/O). Instead, the **same** destination-assignment engine that already evaluates rule 1's promotion condition (§6.1, PR-2) computes `requires_tracking` in the same pass it computes `destination` — one computation, one place, the same discipline §6.2 already applies to `open_items_created`. `requires_tracking` is `true` exactly when `severity: "blocker"`, or `severity: "warn"` at/above the recurrence threshold; `false` otherwise. The validator's job is structural in **two** parts, not one — r2 specified only the first and, on codex's r3 re-review, overclaimed what it covered:

1. **`requires_tracking` ⇒ `destination`** (r2's original check, unchanged): reject any entry where `requires_tracking: true` and `destination` is `"counted"` or `"dropped"`. This is genuinely a trust-the-engine check — it cannot independently re-derive the `warn`-recurred-at-threshold half without the rolling-window read this ADR deliberately keeps out of the append-time validator (above).
2. **`severity` ⇒ `requires_tracking`** (new in this revision, r3 HIGH-3): reject any entry where `severity: "blocker"` and `requires_tracking` is not `true`. Unlike the recurrence half, this is **not** a trust-the-engine check — `severity` is a static field sitting on the same entry the validator is already reading, present with zero rolling-window state required, so the validator enforces this half directly, the same way it already enforces `destination` membership. r2's claim that the matrix "closes the finding completely for `severity: 'blocker'`" was incomplete: check 1 alone only fires when `requires_tracking` is already (correctly) `true` — an engine bug that mis-stamps `requires_tracking: false` on a blocker entry sails past check 1 entirely, because `requires_tracking: true` was never asserted in the first place. `{severity: "blocker", requires_tracking: false, destination: "counted"}` is legal under check 1 alone — `requires_tracking` reads `false`, so the `counted`/`dropped` restriction never applies — exactly the failure scenario r3 HIGH-3 named. Check 2 closes it: independent of what `destination` claims, a blocker entry whose own `severity` field disagrees with its own `requires_tracking` field is rejected on that disagreement alone.

Composed, checks 1 and 2 together make `{severity: "blocker", destination: "counted"}` (or `"dropped"`) unrepresentable in a durable receipt regardless of what `requires_tracking` value the engine attached: either `requires_tracking` is (correctly) `true` and check 1 rejects the `counted`/`dropped` destination, or `requires_tracking` is (incorrectly) `false` and check 2 rejects the severity/tracking mismatch directly. No `requires_tracking` value survives both checks for a `blocker` paired with `counted`/`dropped`. The recurred-`"warn"` case is still closed the way r2 described — not by the validator re-deriving recurrence, but because the one function that *does* derive it (the destination-assignment engine, PR-2) is required to stamp its answer onto the entry, where check 1 can hold it accountable; check 2 does not apply to `warn`-severity entries, since only `severity: "blocker"` is a static, always-`true` case.

| `requires_tracking` | Legal `destination` | Illegal `destination` |
|---|---|---|
| `true` (blocker, or warn ≥ threshold) | `oi`, `oi_pending` | `counted`, `dropped` |
| `false` (warn below threshold, info) | `oi`, `oi_pending`, `counted`, `dropped` (allow-listed reason) | — |

**The validator's full reject list (r1 BLOCKING-3's original scope, extended by r2 BLOCKING-1, r2 HIGH-3, r3 BLOCKING-1, r3 HIGH-2, r3 HIGH-3).** Today's `_validate_receipt` (`scripts/lib/append_receipt_internals/validation.py:53`) checks exactly three keys — `timestamp`, `event_type`/`event` (present, non-empty, aliased unconditionally — §3.2.1), `dispatch_id` — and nothing about `warnings[]` at all; that is expected, since v1 has no `warnings[]` shape yet. v2 extends this validator, and the append it guards **rejects** — raises, writes nothing — on any of:

- `event_type` missing or empty for a `schema_version >= 2` record — the `event` alias is **not** consulted for a v2-shaped record at all (r3 HIGH-2, §3.2.1); for a `schema_version` absent/`1` record, `event_type` **or** its `event` alias must be present and non-empty (unchanged v1 tolerance, r1 BLOCKING-3's original check);
- the resolved event-name value (whichever key supplied it) failing the lightweight format guard `^[a-z][a-z0-9_]*$` — a typo/garbage guard, never a membership check (r3 BLOCKING-1 — this bullet replaces the closed-set `LEGAL_EVENT_TYPES` check revision 2 specified here; see §3.2.1/§11 for why that shape was rejected);
- a `warnings[]` entry missing `destination` entirely;
- `destination` set to any value outside the four legal ones (`oi`/`oi_pending`/`counted`/`dropped`) — a caller-supplied free-form string like `"ignored"` is rejected, not passed through;
- a `warnings[]` entry missing `code`, `severity`, or `requires_tracking`;
- a `warnings[]` entry with `severity` outside `{blocker, warn, info}` (`open_items_manager.py::SeverityLevel`, `open_items_manager.py:38` — the same closed vocabulary today's `_SEVERITY_MAP`, `append_receipt_internals/quality.py:9-13`, already normalizes raw advisory severities into before calling `add_item_programmatic`) — an unrecognized value like `"critical"` is rejected, not passed through (r3 HIGH-3);
- `destination: "oi"` with no corresponding `oi_id` (a promotion claim with no open item behind it);
- `destination: "oi_pending"` with a non-null `oi_id` (the item does not exist yet — a non-null `oi_id` here is a contradiction, not progress) or with `reason: null` (no explanation for why promotion is pending);
- `destination: "dropped"` with `reason: null`, or with a `reason` not drawn from the drop-reason allow-list above (tightened from r1's null-only check — HIGH-3);
- `requires_tracking: true` with `destination` in (`counted`, `dropped`) — the matrix violation r2 HIGH-3's fix targets;
- `severity: "blocker"` with `requires_tracking` not `true` — checked directly against the entry's own `severity` field, independent of `destination` and independent of whatever the destination-assignment engine computed for `requires_tracking` (r3 HIGH-3; see the matrix discussion above for why this is a second, stateless check rather than a restatement of the bullet above).

Concretely: a receipt attempting `warnings: [{"severity": "blocker", "destination": "ignored"}]` is rejected outright — it never lands durably in the ledger. A receipt attempting `warnings: [{"severity": "blocker", "requires_tracking": true, "destination": "counted"}]` is equally rejected on check 1 — the r2 HIGH-3 engine-bug scenario is a rejection, not a silent downgrade. And a receipt attempting `warnings: [{"severity": "blocker", "requires_tracking": false, "destination": "counted"}]` — the exact scenario r3 HIGH-3 named as still legal under check 1 alone, since `requires_tracking` reads `false` — is now also rejected, on check 2 alone, independent of `destination`.

**Binding both write paths (the other half of r1's BLOCKING-3, and r2 BLOCKING-1's Path-1 fix).** Today this validator runs only from `payload.py::append_receipt_payload` (Path 2, `payload.py:348`); `governance_emit.emit_dispatch_receipt` (Path 1) writes the ledger line directly and never calls it — confirmed, `governance_emit.py` has no import of `validation.py` at all, and (§3.2.1) stamps no `event_type` either. A validator that only one of two writers calls is not enforcement, it's a Path-2-only house rule. §7.1 closes this by making the validator (extended with the checks above) run inside the **shared append primitive** both paths call, and by making `emit_dispatch_receipt` stamp `event_type` before it ever reaches that primitive — there is exactly one place a `warnings[]` entry or an `event_type` could be validated, and every write path goes through it with a receipt shape that can actually pass.

### 6.2 `open_items_created` becomes derived

The current `receipt.setdefault("open_items_created", facade._count_quality_violations(receipt))` (a separate dry-run-dedup pass, `quality.py::_count_quality_violations_against_store`) folds into the same destination-assignment engine: `open_items_created` is the count of `warnings[]` entries that resolved to `destination: "oi"` on this receipt — `"oi_pending"` entries are **not** counted here, since no open item exists yet for them; §6.4's reconcile step is what eventually moves a count from "pending" to "created," and it updates the open-items store directly, not this immutable receipt field. One computation, one place, instead of a receipt field and a quality-module dry-run staying in sync by convention.

### 6.3 Migration of the existing mechanism

`quality_advisory.t0_recommendation.open_items[]` is retired as a receipt field name, but its `dedup_key` construction (`qa:{check_id}:{file}:{symbol}`) is preserved verbatim as the `code` field's value for that warning class — an already-tracked open item's `dedup_key` in `open_items_manager`'s store keeps matching a v2 warning describing the same underlying check, so the cutover creates no duplicate or orphaned open items.

### 6.4 `oi_pending` — durable representability when the OI store itself fails (r2 BLOCKING-2)

**Verified gap.** §3.1's `verdict.decision` rule already needed a way to say "a warning is waiting on OI-promotion" — but before this revision, the only legal `destination` values were `oi` (requires a real `oi_id`, which doesn't exist until promotion succeeds), `counted` (silently downgrades a blocker to ambient noise), and `dropped` (asserts the warning was deliberately discarded, which is false — it wasn't discarded, promotion is what's supposed to happen to it). None of the three is honest when `open_items_manager.add_item_programmatic` itself fails. And this failure mode is real, not hypothetical: `_register_quality_open_items` (`quality.py:91-154`, today's single-warning-class precursor to §6.1's general engine) already wraps its `oim.add_item_programmatic(...)` call in a bare `except Exception as exc: _emit("WARN", "quality_oi_item_failed", ...)` (`quality.py:146-147`) — a caught failure today is logged once and the item is silently never created, with no trace left on the receipt itself. `_count_quality_violations_against_store` (`quality.py:37-88`) has the identical pattern one level up (`quality.py:72-74`, `:86-88`). Generalizing this engine to arbitrary `warnings[]` (§6.1, PR-2) without also fixing this failure path would generalize the silent-loss bug along with it.

**The fix: `destination: "oi_pending"`.** When the destination-assignment engine determines a warning must be tracked (`requires_tracking: true` — severity `blocker`, or `warn` at/above the recurrence threshold) and calls `open_items_manager.add_item_programmatic` to promote it, but that call raises or the store is confirmed unreachable/locked, the engine catches that *specific* failure and stamps `destination: "oi_pending"` with `reason` set to the caught exception's message — instead of today's swallow-and-log-only behavior. The receipt still gets appended (§6's opening principle: the writer computes `destination` *before* the append, and the append must never block on OI-store health — see §11's rejected "block-until-success" alternative). `oi_id` stays `null` — the item genuinely does not exist yet, so there is nothing to point at, and stamping a fake or placeholder ID would be a second lie layered on the first.

**The lifecycle: `oi_pending` → `oi` (with `oi_id`) → `resolved`.**

1. **`oi_pending`** — the state on the receipt line itself, permanently. ADR-005 append-only means this line is *never* rewritten once `oi_pending` is durably appended — this is not a temporary label that later flips to `"oi"` in place. The receipt is a historical record of "at write time, promotion could not be confirmed"; that fact stays true forever, regardless of what happens next.
2. **`oi` (with a real `oi_id`)** — not a mutation of the original receipt line, but the *existence of a matching record* in `open_items_manager`'s own store (which is mutable JSON, not the append-only ledger — a different store, a different consistency rule, exactly as ADR-005 already distinguishes runtime/coordination state from the ledger, §4). A reconcile step (below) retries the same `add_item_programmatic` call the original write attempted, using the identical `dedup_key` construction (§6.3's continuity guarantee: `qa:{check_id}:{file}:{symbol}` for the quality-advisory class, generalized per warning `code` for others). Success means a real open item now exists with a real `oi_id`; a reader who wants "is this still pending" joins the receipt-side `oi_pending` warning against the open-items store by `dedup_key`, not by re-reading the (unchanged) receipt line.
3. **`resolved`** — the terminal state of the underlying open item once a human closes it, via the existing `close_item`/`open_items_manager.py` flow (`status: "open"` → `"closed"`/`"deferred"`/`"wontfix"`). No new mechanism needed here; once step 2 produces a real open item, it re-enters the open-items lifecycle VNX already has.

**The follow-up obligation — never a fire-and-forget.** `oi_pending` is a receipt-durable promise, and a promise with no enforcement is exactly the ambient-noise failure mode pillar 4 exists to prevent (§1, pain point 2). Two mechanisms close the loop, both extending machinery this ADR already specifies rather than adding new machinery:
- **Visibility.** `receipt_query.py digest` (§5.2, §9 PR-7) is extended with a second tally alongside its existing `"counted"`-warnings count: "N warnings met `oi_pending` zonder resolutie" — computed the same way as step 2's reader-side join (dedup_key against the current open-items store), so a warning drops out of this count the moment its promotion actually succeeds, without the receipt itself ever changing.
- **Active retry, not passive waiting.** `receipt_query.py reconcile-oi-pending` (§9 PR-7, new subcommand) scans the ledger for `oi_pending` warnings not yet matched in the open-items store and retries `add_item_programmatic` for each, using the preserved `dedup_key`. This is designed to be run from the same pull cadence T0 already uses (§5) — a routine part of a T0 cycle, not a separate operational task someone has to remember to invoke. An entry that keeps failing past a configurable retry/age threshold escalates rather than retrying forever — surfaced the same way any other standing operator obligation is (T0 digest, not a new alerting channel).

Table T31/T34 (§10) are the regression guards: T31 for the receipt-side legality of `oi_pending`, T34 for the reconcile-then-drops-out-of-the-tally behavior.

## 7. Ledger hygiene and compatibility

- **Append-only preserved (ADR-005).** No v1 line is ever rewritten. v2 receipts are new lines appended from the cutover point forward; a ledger is a v1-shaped prefix followed by a v2-shaped suffix, mirroring the pattern ADR-029 already established for the hash-chain's unchained-epoch-0-prefix-then-chained-suffix — mixed-shape ledgers are not a new concept for VNX readers, and every reader in §9's PR list is written version-aware from the start, never assuming uniformity.
- **`schema_version` is the version tag.** Field presence (not a file-header marker), consistent with how `prev_hash` (ADR-023) and `chain_epoch_start` (ADR-029) already signal "this capability is active from here forward" without touching history. Absent or `1` means legacy v1 shape; readers treat that as `verdict: "unknown"`, never backfilled.
- **Hash-chain/epoch/anchor *mechanics* (ADR-029, ADR-034) are untouched by this ADR.** `ndjson_hash_chain.py`'s `verify_chain` hashes the serialized JSON line's byte content, not a fixed field set — it has already absorbed one schema change mid-ledger before (`events_path`, PR #843, ADR-005 amendment) without any chain-side accommodation. A v1→v2 payload-shape transition, including mid-open-epoch, is the same class of change; this ADR does not modify `ndjson_hash_chain.py`, `chain_epoch_seal.py`, or `chain_origin_anchor.py` (design-only, ADR-034), and does not flip `VNX_CHAIN_RECEIPTS`. T19 (§10) is the regression guard proving this. What this ADR *does* change is which code stamps the per-line `prev_hash` chaining field that `verify_chain` reads — §7.1, closing BLOCKING-1.
- **Receipts stay local — hard constraint, restated (ADR-026).** Nothing in this ADR ships receipt content to a cloud store. `receipt_query.py`'s pull/digest interface reads `~/.vnx-data/<project_id>/state/t0_receipts.ndjson` on the same host; no network call is introduced.
- **Both write paths gain v2 support through one shared primitive, not two parallel fixes.** See §7.1 — this is the mechanism, not just the intent, behind "symmetric handlers" (Codex Defense Checklist: "same fix to all handlers").

### 7.1 The shared append primitive — closing the hash-chain writer-contract gap (BLOCKING-1) and binding enforcement to both paths (BLOCKING-3)

**Verified gap.** `docs/core/11_RECEIPT_FORMAT.md:96` already states this plainly: "`emit_dispatch_receipt` (Path 1) does NOT honor `VNX_CHAIN_RECEIPTS`; enabling the flag on a deployment that uses both paths produces a mixed ledger that verifies as `broken`." Reading both write paths confirms why, and that it is worse than a missing `if`-check in one function:

- Path 2 (`append_receipt_internals/idempotency.py::_write_receipt_under_lock`, lines 172-215) takes an exclusive `flock` on a **dedicated lock file**, `append_receipt.lock` (`_lock_file_for`, line 81-82), and — only inside that lock, only when `_chain_receipts_enabled()` is true — reads the ledger's tail hash and stamps `receipt["prev_hash"]` **before** appending. The read-tail-then-stamp-then-write sequence is fully serialized under that one lock; that serialization, not just the presence of the stamp, is what stops concurrent Path-2 writers from forking the chain.
- Path 1 (`governance_emit.py::emit_dispatch_receipt`, lines 141-152) opens the **receipts file itself** (`t0_receipts.ndjson`) and `flock`s that file handle directly — a **different lock, on a different inode**, from Path 2's `append_receipts.lock`. It never calls `_chain_receipts_enabled()` and never stamps `prev_hash` at all.

Two consequences follow, and only the first is what BLOCKING-1's finding named directly: (1) a Path-1 write appends an unchained line into a chaining-active ledger, which `verify_chain` (`ndjson_hash_chain.py:278-290`) reports as `"broken"` outright; (2) **fixing only (1)** — e.g. patching `emit_dispatch_receipt` to also call `_chain_receipts_enabled()`/stamp `prev_hash`, using its *own* existing lock — would still be unsound, because Path 1's lock and Path 2's lock do not serialize against each other. A Path-1 write and a Path-2 write landing concurrently could each read the same tail hash under their own, different locks and fork the chain even with both paths "honoring" the flag. The writer-contract this ADR specifies has to be about the *lock*, not just the *field*.

**The fix: one call site, not two parallel implementations.** `_write_receipt_under_lock` (Path 2's existing chain-aware, lock-file-based writer) becomes the **only** code path that opens, locks, and appends to `t0_receipts.ndjson`. `emit_dispatch_receipt` (Path 1) stops doing its own inline `open()`/`flock()`/`write()`; it still builds its receipt dict from the same inputs it does today (`dispatch_id`, `status`, `token_usage`, etc.), plus one addition — `event_type: "task_complete"` (§3.2.1, r2 BLOCKING-1: the field the shared primitive's validator is about to start requiring, on the one call site that has never set it) — then calls the shared primitive to perform the actual append. This is not a rename of two functions to look similar — it is deleting Path 1's independent write implementation and routing it through Path 2's, with the one dict-construction change Path 1 needs to survive contact with a validator it never faced before. Path 1 gets, by construction rather than by two authors remembering to keep two implementations in sync:

1. **The same lock file** as Path 2 — a concurrent Path-1 and Path-2 write now genuinely serialize against each other.
2. **The same `_chain_receipts_enabled()`/`_last_hash_under_lock()`/`prev_hash` stamping** — closing BLOCKING-1 completely, not partially. No v2 write path can append an unchained line into a chaining-active ledger; this is true because there is exactly one append code path, not because two append code paths were each separately taught the rule.
3. **The same validator call site** — the extended `_validate_receipt` (§6.1's reject list) runs for every receipt, from either path, before the write commits. This is what makes BLOCKING-3's "bind enforcement to both write paths" true by construction rather than by a second, parallel validation call added to `governance_emit.py`.
4. **A receipt shape that can actually pass that validator** — since (3) now applies to Path 1 for the first time, Path 1 must also gain whatever the validator has always required. `event_type` is the one gap: the dict-construction addition above closes it in the same PR that exposes Path 1 to the validator, so there is no window where PR-3 ships "Path 1 now validated" and Path 1 is still building an event-type-less dict that the validator it's now subject to would reject on sight (r2 BLOCKING-1's exact failure scenario).

No new lock file, no new chain primitive, no new validator: the fix is consolidation onto what Path 2 already has, plus the one field Path 1 was always missing, not new machinery. §9 PR-3 is this extraction; T21 (§10) is the concurrent-writer regression guard for point 1, T30 (§10) is the regression guard for point 4.

## 8. Non-goals

- **Not rebuilding the markdown report format.** `verification{}` reuses the existing `report_parser.py::extract_validation` regex extractor; it does not mandate a new report structure. Structured JSON reports remain their own horizon track (`structured-json-reports`, plan-gated) per the explicit 2026-07-09 finding that the 2274x number is dead noise, not a live defect to architect around.
- **Not resurrecting `feat/receipt-mailbox-delivery` as a branch.** Its cursor *algorithm* is absorbed (§5.1); the 1,509-commits-stale branch itself is not merged or rebased.
- **Not touching `VNX_CHAIN_RECEIPTS`'s semantics, epoch rotation, or the chain-origin anchor.** ADR-029 and ADR-034 stand exactly as designed — the flag means what it always meant, and `ndjson_hash_chain.py`/`chain_epoch_seal.py`/`chain_origin_anchor.py` are not modified. §7.1 changes *which code path* stamps `prev_hash` when the flag is on (unifying two independent writers onto one), not what the flag does or how the chain is verified.
- **Not introducing a SQLite projection or index for receipt queries in this phase.** Linear NDJSON scan is the mechanism for `by-dispatch`/`by-pr`/`since`/`pull`; `by-track` is the one exception, and it isn't a new projection — it reuses the existing `dispatches.track` relation and the existing `tracks.get_recent_receipts` query shape (§5.2), not a new index built for this ADR. A dedicated projection DB for the others is future work gated on measured scan-cost need, not assumed here.
- **Not adding `track_id` as a receipt field.** Rejected for the same reason `session{}` was cut in §4: it is `dispatches.track`-owned relational state, not a fact about the dispatch outcome. §5.2 joins through the existing DB relation at query time instead.
- **Not backfilling or rewriting v1 receipts.** ADR-005 append-only is absolute.
- **Not removing `orchestrator_id`/`agent_id`**, or any other identity/attribution field — out of scope for this pass; see §3.3's explicit distinction from terminal/session *state*.
- **Not removing `VNX_RECEIPT_T0_PUSH` in this PR.** It remains the transition escape hatch until pull is confirmed wired fleet-wide (§5.3); removing it is a documented follow-up.
- **Not shipping receipts, or any receipt content, to Supabase or any cloud store** (ADR-026 hard constraint, restated).

## 9. PR decomposition (150–300 lines each, implementation dispatch)

**Reordering rationale (HIGH-6).** The earlier draft stamped `schema_version: 2` in PR-3, before PR-4's field rename/trim had landed — a deployment between those two merges would produce receipts labeled `"2"` that still carried the v1 field names (`validation{}`, `session{}`), indistinguishable from true v2 shape to any reader keying off the version number alone. The fix is sequencing, not a new marker type: every PR that only **adds** fields on top of the current (v1) shape ships without touching `schema_version` at all — additive fields are always safe for existing readers, which already tolerate unknown keys (`_validate_receipt` inspects exactly three keys, never rejects on extras). `schema_version: 2` is stamped in exactly one PR — the one that also performs the last destructive shape change (rename + trim) — so the version number and the complete shape become true in the same commit, never separately.

## 11. Alternatives considered and rejected

- **SQLite-first receipts.** Already settled by ADR-005; not reconsidered here. NDJSON stays canonical, SQLite (if ever needed for query performance) stays a downstream projection.
- **Backfilling v1 records to the v2 shape.** Violates ADR-005 append-only immutability outright. Rejected structurally, not on preference.
- **LLM-computed verdict** (a model reads the report and states accept/investigate/reject). Rejected: non-deterministic, not unit-testable, and not auditable in the way a fixed rule table over `status`/`verification`/`warnings[]` is. The whole point of `verdict{}` is that it is mechanically reproducible from the receipt's other fields.
- **A new index/SQLite projection to back `receipt_query.py`'s PR/track/timestamp lookups.** Premature at current per-project ledger scale (~8k–13k lines); linear scan is what `receipt_provenance.py` already does successfully for dispatch-indexed lookups. Revisit only if scan cost is measured as a real bottleneck.
- **Renaming `provenance` to `diff` for clarity.** Rejected once `receipt_provenance.py::find_receipt_by_commit`'s live dependency on `provenance.git_ref` was found — a naming preference does not outweigh breaking a working consumer for free. The field keeps its name; only its dead subkeys are trimmed.
- **Folding `orchestrator_id`/`agent_id` into the terminal-state cleanup.** Considered and rejected: those are attribution facts (stable for the dispatch's life), not the mutable environment-state class (`session{}`, `observability_tier`) this ADR's pillar 2 targets. Conflating the two would widen this ADR's blast radius without evidence it's part of the operator's complaint.
- **Resurrecting `feat/receipt-mailbox-delivery` wholesale.** Rejected: 1,509 commits stale, predates the #1178 push-hardening this ADR's §5.3 explicitly reconciles with. The validated *design* (cursor pull) is kept; the branch is not.
- **Patching `emit_dispatch_receipt` to stamp `prev_hash` under its own existing lock, without unifying the write path.** Rejected (BLOCKING-1): Path 1's lock (on the receipts file handle) and Path 2's lock (a separate `append_receipt.lock` file) are two different advisory locks. Making Path 1 chain-aware in isolation closes the "unchained line" symptom but not the underlying race — a concurrent Path-1 and Path-2 write could still read the same tail hash under their own locks and fork the chain. Only a single shared lock (§7.1) closes this completely.
- **Adding `track_id` as a receipt field.** Rejected (HIGH-4): reintroduces the exact anti-pattern §4 already removed for `session{}` — duplicating DB-owned relational state (`dispatches.track`) onto the immutable ledger. A SQLite join at query time, reusing `tracks.get_recent_receipts`' existing query, is both more consistent with pillar 2 and the only option that doesn't require every future receipt to carry a field solely to make one query subcommand work.
- **Dropping `quality_advisory{}` in the same PR that introduces `warnings[]`.** Rejected (HIGH-5): `cqs_calculator.py` reads `quality_advisory.t0_recommendation` directly for 10% of its CQS score; a same-PR swap would silently zero that component the moment the PR merged, with no test failure to catch it (the consumer isn't in this ADR's own test list unless added). The consumer migrates first (PR-4b), verified by parity test T28, *then* the field is dropped (PR-5).
- **Treating `recommendations`/`quality_context`/`pattern_count`/`missing_fields`/`tags` as one dead-weight bucket, as an earlier draft of this ADR did.** Rejected on re-verification: four of the five have real, executable readers (§3.2); bucketing them with genuinely dead fields (`root_cause`, `metrics{}`, etc.) would have silently degraded T0-recommendation generation, intelligence-health coverage reporting, and a receipt-backfill utility. Every field in the final §3.3 removed-table is now individually grepped, not pattern-matched against a sibling.
- **Stamping `schema_version: 2` as soon as `verdict{}`/`warnings[]` land (the earlier PR-3), ahead of the field rename/trim.** Rejected (HIGH-6): produces a window where receipts are version-labeled "2" but shaped like v1.5 (`validation{}`/`session{}` still present), indistinguishable to a reader keying off the version number. The additive PRs now ship unversioned; the version number and the final shape land together, in one PR (§9 PR-5).
- **Softening `_validate_receipt`'s `event_type` requirement instead of fixing `emit_dispatch_receipt`.** Rejected (r2 BLOCKING-1): the finding's own failure scenario is that PR-3 routes Path 1 through the shared validator while `emit_dispatch_receipt` still builds an event-type-less record — relaxing the validator to tolerate that would make Path-1/v2 receipts durably event-less, silently breaking every consumer that filters on `event_type == "task_complete"` (`outcome_signals.py`, `pr_discovery.py`, `unified_state_manager.py`). The append primitive being "hot-path-safe" means the writer meets the validator's existing bar, not the other way around. Fixed at the writer (§7.1 point 4), not the gate.
- **A closed-set allow-list (`LEGAL_EVENT_TYPES`) for `event_type`, checked by membership.** This was this ADR's own r2-revision design, and codex correctly flagged it as fragile on re-review (r3 BLOCKING-1): a fresh grep for this revision turns up 50+ distinct `event_type` string literals already written across the tree — governance events (`emit_governance_receipt`), report-contract/gate events (`report_contract_invalid`, `pr_merged`, `phantom_guard_error`, `context_rotation_continuation`), and more — with no central registry constraining what a future PR adds. A closed enum against an evolving, ungoverned value space doesn't fail once at design time; every new event type a future PR introduces would find its receipts silently rejected by a validator this same ADR promised would stop silent receipt loss — a self-defeating outcome for pillar 1. Required-and-non-empty, plus a lightweight format guard that only catches transcription garbage (never a novel-but-valid value), gets BLOCKING-1's real target — no event-less governed receipts — without freezing the event vocabulary. A genuine future need for a closed vocabulary (e.g. to catch a specific class of typo bug, empirically observed) is a narrower, evidence-driven follow-up, not a speculative allow-list adopted up front.
- **Making the `event_type`/`event` alias universal across schema versions, with no `schema_version` gate.** Rejected (r3 HIGH-2): simplest to implement, but it reopens the exact gap r2 BLOCKING-1 closed for Path 1's *missing* key — a v2 write could carry only `event` and still pass validation, landing durably invisible to every consumer that filters on `event_type`. Gating the alias on `schema_version` costs one conditional branch and preserves both properties at once: v1 lines already on disk (and any v1-shaped write during the rollout window) keep validating exactly as before, and v2 writes are held to the field name every v2 consumer actually expects.
- **Trusting `requires_tracking` completely for the `severity: "blocker"` case, the same way the validator must for the recurred-`"warn"` case.** Rejected (r3 HIGH-3): the two cases are not symmetric. The recurred-`"warn"` half genuinely needs rolling-window state the append-time validator doesn't have access to (§6.1's existing reasoning, unchanged by this revision) — trusting the engine there is a real, unavoidable trade-off. But `severity` is a static field sitting on the same entry the validator is already reading; checking `severity: "blocker" ⇒ requires_tracking: true` directly requires no I/O, no rolling window, and no scope-widening of the shared append primitive's lock (§7.1). Extending the "trust the engine" reasoning to a case that doesn't require trust would leave exactly the engine-bug scenario r3 HIGH-3 named (`requires_tracking: false` stamped on a blocker) unrejected, for no reason beyond superficial consistency with the one case where trust actually is necessary.
- **Silently downgrading a blocker's `destination` to `"counted"`/`"dropped"` when `open_items_manager` fails.** Rejected (r2 BLOCKING-2): this is receipt-content lying about what happened — a blocker that should be tracked reads as ordinary aggregate noise, indistinguishable from a real low-priority warning in `digest`'s counts. `oi_pending` (§6.1, §6.4) keeps the failure visible and durably attributable instead of erasing it.
- **Blocking the receipt append entirely until `open_items_manager` succeeds (retry-until-success inside the write path).** Rejected (r2 BLOCKING-2, implicit alternative): would trade "the OI might not exist yet" for "the receipt might not exist yet either," reintroducing exactly the receipt-loss risk this whole ADR's pillar 4 exists to close, and adds unbounded I/O (a lock-contended store retry loop) inside the shared append primitive's lock scope (§7.1) — a scope this ADR deliberately keeps narrow (hashing + writing, no arbitrary I/O). The receipt must land regardless of OI-store health; `oi_pending` plus an out-of-band reconcile (§6.4) is what makes that consistent with "every blocker still gets tracked, eventually."
- **Free-form `reason` text for `destination: "dropped"` (r1's original rule — non-null is sufficient).** Rejected as of this revision (HIGH-3): a non-null-but-arbitrary reason ("meh", "not important") is functionally identical to no enforcement at all — anyone can write a plausible-sounding excuse at append-time. A closed allow-list (§6.1) makes "why was this dropped" a reviewable, finite question instead of free text nobody re-reads.
- **Letting the validator recompute `severity: "warn"` recurrence itself, instead of trusting a `requires_tracking` field the writer stamps.** Rejected (HIGH-3): would require the validator to read rolling-window history (other receipts, or the open-items store) from inside the shared append primitive's lock — new, unbounded I/O in a scope this ADR keeps narrow (see the retry-until-success rejection above), and a second, independent implementation of the exact recurrence logic PR-2's destination-assignment engine already computes, risking the two silently disagreeing. `requires_tracking` is computed once, by the one function with the data, and carried on the entry for a cheap structural check.
- **Fail-open on doc-only when `provenance.diff_summary.paths` is missing (`method: "n/a"` with no `paths` defaults to `accept`).** Rejected (r2 HIGH-4): this is exactly the accept-without-evidence gap the doc-only invariant exists to close, just moved one level down — absence of `paths` is absence of evidence, not evidence of doc-only-ness. `investigate` is the only fail-safe default (§3.1.1).
- **Leaving `provenance.diff_summary.paths` population in PR-5 and accepting the PR-4→PR-5 gap as a documented, temporary known-issue.** Rejected (r2 HIGH-4): this ADR already rejected the equivalent trade-off once, for `schema_version` (HIGH-6, r1) — a "temporary, documented" incorrect-shape window is still a window where a real dispatch's receipt gets evaluated against a rule that can't do its job. The fix costs one PR reordering, not a new mechanism; there is no reason to accept the gap when the fix is this cheap.
